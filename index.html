Introduction

This website tells the story of how I came to implement VenStipe. I'll explain what it is, or at least, supposed to be, in due time because, when this story started, I was nowhere near think of it. I hope this project sees it's implementation finished, one day. Finished or not, I will tell its whole story here. From a certain point on, that I'll mention, as it happens.

The Sparkles

I won't go into the the discussion of what is original or if original actually exists but I'll list the sequence of events, feelings and ideas that led to me wanting to implement VenStipe.

The connected dots

There were three major factors that set me on this path:

  • I like time waster games

    I like time waster casual games, mostly those that require brain usage providing, what I call, an active rest. Games like:

    • 2048
    • Flow
    • Flow: Bridges

    Anyone can find me playing those on my work breaks, on the bus or while walking on the street by myself.

  • I like software development

    I make a living as a software developer. I've been doing so since 2008. That story is told here. For this story, it's important to note that: I'm very fond of generics and sharing funcionality, i.e., implementing things in a one-size-fits-all, non-functionally-repetitive fashion; I try (almost) everything that crosses my path and regards to software development.

  • I don't like repeating completed things

    Being a technology person, I regularly change device and platform, test new operating systems, and so on. It solemnly annoys me that, every time I change something, all progress is lost: high scores in 2048 and levels complete in Flow and Flow: Bridges. My highest score in 2048 6x6 took me about two months.

Time wasting and curiousness

I was playing 2048 and strolled to the about section that says:

Based on 2048 by Gabriele Cirulli
...and provided a link to the GitHub.io page of the original game. My curious self visited that page and that page pointed me to the GitHub repository where the source code of the game was. It's implemented in JavaScript and so, the game I was playing, is for sure (although I didn't and won't make any effort to confirm), implemented using something like Cordova that, basically, generates an application that opens a web browser and runs the web page in it.

A sequence of events

2015 Late December

I decided to port 2048 to C#. Along the way, I decided to put some use to my GitHub account and start sharing something on it. So I created my first GitHub repository, 2048.net, and ported the JavaScript code to C#. I let it snooze for a while. I created a WPF project to use it but that wasn't a very sharable platform.

I had used Xamarin Forms, professionally, but, althgough its first version was, as first versions usually are, quite limited in funcionality, one could see that its creation was a big step in the right direction. When version 2 came out I had to give it a go. It's always hard to come up with ideas, at least, good ones, to test a technology. So, I had my C# port of 2048 parked and I wanted to try out Xamarin Forms. The result was an implementation of the game in Xamarin Forms that I made sure worked on iOS, Android and Windows 10 desktop, Mobile and IoT on a Raspberry Pi 3 I had just bought "for the kids".

I implemented the gameplay and storing game state and high scores locally, as the JavaScript version does, but, after that, I had to implement a service to store those remotely so that high scores, first, and game state, after, would be shared cross instances of the game. I.e, when I stop playing in my iPad and, afterwards, I pickup playing in my Raspberry Pi, I'm exactly in the same point in the game.

The idea

As I said before, I'm very fond of implementing things in a sharable way so, in this case, I would create a service that all games I implemented could/would use. But, if I was to implement a service for my games to use, why not implement a service that all games could use?

It was clear that I had to implement such a service that all game developers could (easily) use to provide such funcionalilty as game state (e.g., high scores and progress) storing to share cross-instances of the game.

The Start

Brainstorming

2016 Late March

The idea was born: a service that provides storing of game states for all game developers. I, immediately, discussed the idea with two very close friend developers. I can track the first Skype chat about VenStipe to the 29th March 2016. The more I discussed the purpose and of the project and the tons of things that could (will) be implemented around it, the more I was convinced I had to do it.

I did some research on existing services and found that there isn't one that is platform agnostic and, in more conversations, more ideas on what to do around the service came about, like:

  • List high scores for a specific game particularly
  • Notifications of high scores beaten
  • Host tips, tricks and solutions
  • Integration with existing services
  • ...and many more...

Overview

2016 May, the 4th

As a developer, I tend to make the mistake of putting my hands to the task before putting everything into documentation and, thus, settle how and what things should be. I created a VenStipe repostory in my own GitHub account and created a Visual Studio solution with the projects, at the time, I thought were neeeded for the whole service:

  1. An objects definition library
  2. A Web API project that would talk to a database in the backend
  3. A Web Site with two faces: for game developers; for gamers
  4. A PCL client of the Web API for game developers to use

This was a very productive day. In the same day, I had a discussion with my two fellow friend developers on how to call the service. There were two ideas I wanted reflected in the name, somehow: game and gathering. I had, from start, the idea of using latin as the base for the name although I looked at the word game in other languages (actually, all Google Translate supported, at the time). Here is a list of some possible names that came by:

  • oneludos
  • omniludos
  • allludos
  • alludos
  • gamestipe
  • ludoszone
  • ludozone

VenStipe comes from the latin words venatus, that means game (or hunt) and Stipes, that means club. Later, I realized Stipes means club as batting club or bat and not club as an association. But, when that happen, the choice was already made and the domain was already registered. The name is short easy to pronounce and to remember.
